# Performance

Tracking error, disturbance rejection, noise rejection, bandwidth, overshoot, control effort.

## What It Means

**Performance** is a system properties topic in the **System Analysis** branch of the control-theory map. In this context, it focuses on structural facts such as controllability, observability, passivity, sensitivity, and performance limits. It belongs to analysis methods that explain stability, robustness, performance, safety, and structural limitations, so the important question is not only what the method computes, but what assumptions make the computation meaningful for a closed loop.

A useful way to read this topic is as a connection between three objects: a model of the plant or signal, an objective stated in engineering terms, and evidence that the resulting behavior is stable, accurate, robust, or safe enough for the application.

## Mathematical Form

A useful control-theory abstraction is a dynamical map with an objective:

$$\dot x=f(x,u,d,t),\qquad y=h(x,u,t),\qquad e=r-y.$$

The notation should be interpreted with the page topic in mind: $x$ is the state or configuration, $u$ is the manipulated input, $y$ is the measured output, $r$ is a reference, $e$ is an error signal, and disturbances or modeling errors are included when they materially affect the design.

## Intuition

The key intuition is to ask what information this topic adds to the feedback loop. Some topics add a model, some add a controller, some add an estimator, and some add a test. For **Performance**, the central issue is the behavior summarized above: tracking error, disturbance rejection, noise rejection, bandwidth, overshoot, control effort. The method still has to remain consistent with the plant assumptions and the performance objective.

A practical example is an engineering design review: the topic helps decide what model is trusted, what signals are measured, what behavior is required, and what evidence is needed before implementation.

## How It Is Used

1. Specify the signals, units, sampling convention, operating point, and uncertainty assumptions.
2. Choose the representation that matches the topic: transfer function, state-space model, graph, stochastic model, optimization problem, or certificate.
3. Apply the method to obtain a controller, estimate, trajectory, reduced model, stability statement, or diagnostic quantity.
4. Check the result against the original engineering requirement, including actuator limits, disturbances, noise, and implementation constraints.
5. Validate with an independent calculation, simulation, experiment, or residual check before relying on the conclusion.

## Common Pitfalls

- Define assumptions explicitly: time domain, sampling, uncertainty, measured outputs, actuator limits, and disturbances.
- Verify conclusions with simulation or analysis that matches the intended closed-loop interconnection.
