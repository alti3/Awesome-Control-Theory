## What It Means

**Trajectory optimization** is a optimal planning topic in the **Planning** branch of the control-theory map. In this context, it focuses on optimization-based generation of feasible state and input histories. It belongs to planning methods that create references, paths, trajectories, or constraint descriptions for downstream feedback control, so the important question is not only what the method computes, but what assumptions make the computation meaningful for a closed loop.

A useful way to read this topic is as a connection between three objects: a model of the plant or signal, an objective stated in engineering terms, and evidence that the resulting behavior is stable, accurate, robust, or safe enough for the application.

## Mathematical Form

The continuous-time optimal-control template is

$$
\min_{u(\cdot)} J=\phi(x(T))+\int_0^T L(x(t),u(t),t)\,dt\quad\text{s.t.}\quad \dot x=f(x,u,t).
$$

The notation should be interpreted with the page topic in mind: $x$ is the state or configuration, $u$ is the manipulated input, $y$ is the measured output, $r$ is a reference, $e$ is an error signal, and disturbances or modeling errors are included when they materially affect the design.

## Intuition

The key intuition is to ask what information this topic adds to the feedback loop. Some topics add a model, some add a controller, some add an estimator, and some add a test. For **Trajectory optimization**, the central issue is the behavior summarized above: searches over state and input histories that satisfy dynamics and minimize cost. The method still has to remain consistent with the plant assumptions and the performance objective.

A practical example is autonomous motion: a planner proposes a feasible route or trajectory, then a tracking controller or MPC layer respects actuator and safety limits while correcting model error.

The G-FOLD trajectory optimisation algorithm is a useful aerospace example. Powered-descent trajectory shaping is naturally non-convex: thrust magnitude lower bounds, thrust-pointing limits, glide-slope constraints, mass variation, and terminal landing requirements make the feasible set hard to search reliably onboard. G-FOLD used **lossless convexification** to relax the non-convex powered-descent optimal-control problem into a convex optimisation problem, while proving that the relaxed solution still satisfies the original non-convex constraints under the problem assumptions. That converts a fuel-optimal landing guidance problem from a brittle nonlinear search into a second-order cone program that can be solved fast enough for real-time guidance.

## Common Pitfalls

- Test solver convergence, infeasibility handling, scaling, and warm starts at the intended real-time rate.
- Distinguish geometric feasibility from dynamic feasibility and from closed-loop trackability.
