# Step input

Tests tracking, settling behavior, overshoot, and steady-state error.

## What It Means

**Step input** is a reference inputs topic in the **Planning** branch of the control-theory map. In this context, it focuses on standard test commands used to expose dynamic response, tracking error, and frequency behavior. It belongs to planning methods that create references, paths, trajectories, or constraint descriptions for downstream feedback control, so the important question is not only what the method computes, but what assumptions make the computation meaningful for a closed loop.

A useful way to read this topic is as a connection between three objects: a model of the plant or signal, an objective stated in engineering terms, and evidence that the resulting behavior is stable, accurate, robust, or safe enough for the application.

## Mathematical Form

Planning usually searches for a feasible curve or graph path:

$$q_{0:N}^*=\arg\min_{q_{0:N}}\sum_{k=0}^{N-1} c(q_k,q_{k+1})\quad\text{s.t.}\quad q_k\in\mathcal C_{\mathrm{free}}.$$

The notation should be interpreted with the page topic in mind: $x$ is the state or configuration, $u$ is the manipulated input, $y$ is the measured output, $r$ is a reference, $e$ is an error signal, and disturbances or modeling errors are included when they materially affect the design.

## Intuition

The key intuition is to ask what information this topic adds to the feedback loop. Some topics add a model, some add a controller, some add an estimator, and some add a test. For **Step input**, the central issue is the behavior summarized above: tests tracking, settling behavior, overshoot, and steady-state error. The method still has to remain consistent with the plant assumptions and the performance objective.

A practical example is a motor-speed or temperature loop: the model predicts the dominant time constants, the controller sets bandwidth and damping, and tests such as step response or frequency response reveal whether the design meets overshoot, settling, and disturbance-rejection targets.

## How It Is Used

1. Specify the signals, units, sampling convention, operating point, and uncertainty assumptions.
2. Choose the representation that matches the topic: transfer function, state-space model, graph, stochastic model, optimization problem, or certificate.
3. Apply the method to obtain a controller, estimate, trajectory, reduced model, stability statement, or diagnostic quantity.
4. Check the result against the original engineering requirement, including actuator limits, disturbances, noise, and implementation constraints.
5. Validate with an independent calculation, simulation, experiment, or residual check before relying on the conclusion.

## Common Pitfalls

- Distinguish geometric feasibility from dynamic feasibility and from closed-loop trackability.
- Define assumptions explicitly: time domain, sampling, uncertainty, measured outputs, actuator limits, and disturbances.
